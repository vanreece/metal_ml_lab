## Operational rules per chip (cheat sheet)

### M1 Pro / `applegpu_g13s`

- Dispatch-overhead floor: ~8 µs back-to-back (`write_tid` 32t).
- Per-chip "nightmare zone": **avoid sleep_1ms** for any single-
  kernel microbench (cv = 7.0, bimodal).
- Warmup recipe: K=1 of the **same** kernel kind right before each
  measurement. Avoid `fma_loop K=1` (cv 1.4-3.8 — destabilizer).
- Pair timing: useful for relative-magnitude / tail-suppression
  questions, NOT for variance reduction (42 µs inter-encoder gap
  breaks the cancellation). Decision 004.
- Work-dominance thresholds: write_tid ≥ 131K-262K threads,
  fma_loop ≥ 256 iters (with caveat about the +21 µs step at
  192→256).

### M4 Max / `applegpu_g16s`

- Dispatch-overhead floor: ~6.4 µs back-to-back (`write_tid` 32t).
- No "nightmare zone" — sleep_1ms is the cleanest cadence (cv=0.07).
- Warmup recipe: K=1 of `heavy_write` (write_tid 1024t) is the
  safest cross-cadence default. `fma_loop K=1` is also safe.
  **Avoid `same K=1`** at sleep_10ms-100ms (it inverted from M1 Pro
  and is now the destabilizer).
- Pair timing: **primary methodology** (decision 005) for trials
  ≥ ~64 µs (≥ 10× the dispatch-overhead floor). Within-session
  variance reduction works for noisy trials; cross-session ratios
  stable to ≤ 0.6 %.
- Work-dominance thresholds: write_tid ≥ 524K threads (~2× shift
  from M1 Pro), fma_loop ≥ 256 iters (same +13 µs step at 192→256).
- Avoid fma_loop iters 4K-16K for between-run-comparable
  measurements (bimodal band, slightly broader than M1 Pro's
  8K-16K band).

## Telemetry stack

Two collectors, both write CSV with `time.monotonic_ns()` for
post-hoc joining with experiment data:

| tool | location | sudo? | what it gives | when to use |
|------|----------|------:|---------------|-------------|
| `gpu_telemetry.py` | `notes/` | yes | powermetrics-derived: GPU active residency %, GPU active frequency, GPU power, thermal pressure | absolute power claims with tight tolerance; cross-validation |
| `ioreport.py` | `notes/` | **no** | libIOReport-derived: 11 power buckets (GPU, CPU, ANE, DRAM, AMCC, DCS, AFR, DISP, ISP, AVE, GPU SRAM) + per-bucket `_energy_nj` columns | default for any new experiment; subprocess from inside `run.py` |

Calibration notes (per exp 008):
- IOReport GPU power agrees with powermetrics to ≤ 5 % across most
  of a 0→100 % busy staircase, but reads **+14 % higher at full
  saturation** and -20 % lower in the post-workload tail. Mechanism
  not isolated.
- IOReport's bonus columns (CPU / DRAM / AMCC / etc.) have no
  powermetrics counterpart at this granularity — they're free
  signal we didn't have before.
- Driving the GPU at ~1 ms/dispatch from Python burns ~2.5 W of
  CPU at peak (not free).

Pattern for new experiments wanting telemetry:

```python
ioreport_proc = subprocess.Popen(
    ["uv", "run", str(NOTES_DIR / "ioreport.py"),
     "--interval-ms", "1000",
     "--csv", str(out_dir / f"{ts}-ioreport.csv")],
    stdout=open(log_path, "w"), stderr=subprocess.STDOUT,
)
# ... run workload ...
ioreport_proc.send_signal(signal.SIGINT)
ioreport_proc.wait(timeout=5)
```

See `experiments/008-ioreport-vs-powermetrics-power/run.py` for
the full reference implementation.

## What we learned about the agent-first lab discipline today

- Per-chip decisions are now the norm (decisions 004 + 005 split
  on chip). Future decisions should default to per-chip scope
  unless they explicitly claim cross-chip validity AND have
  multi-chip evidence.
- The agent-research synthesis approach (delegate broad surveys to
  a sub-agent, get a sourced report) is faster than reading docs
  ourselves but **not always right**: the survey claimed ioreg
  utilization was a sudo-free substitute for powermetrics
  active residency; exp 007 falsified that. Worth doing the
  survey, worth verifying every load-bearing claim with a
  pre-registered experiment.
- Negative results pile up cheaply when pre-registered: exp 007
  produced a clean falsification in ~60 s of GPU work + ~10 min
  of writeup. Worth more than a vague "we tried it and it didn't
  work" because it specifies the mechanism (driver-side update
  cadence too sparse) and tells future-us not to retry the same
  approach.
- The two-terminal coordination dance (user runs sudo collector,
  Claude runs experiment) is workable but the friction adds up.
  IOReport via subprocess (validated by exp 008) eliminates the
  dance for any future experiment that's OK with the +14 %
  full-load bias.
